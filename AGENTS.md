# NixOS Dotfiles

## Critical Non-Obvious Behaviors

### Rime Input Method Data (`rime-sync/nixos-ysun/`)

These files are auto-generated by the input method and change frequently. Always commit them as-is alongside the main work of the turn; do not stop to ask whether they should be included, and do not mention them in the commit message unless the user explicitly asks. Conflicts are expected and usually trivial: keep the newer timestamp.

### Nushell Proxy Wrapper

`claude-proxy` wraps Claude Code and injects proxy environment variables from `~/.config/claude/proxy.nuon`. If network access looks broken, check `home/shell/nushell/config.nu`.

### Upstream / Downstream Boundary

This repo is the reusable upstream base for personal machines and downstream
consumers such as `/home/ysun/github.com/sctmes/dotfiles`. Preserve that
abstraction when changing shared modules, profiles, or flake outputs.

- Expose reusable behavior through stable flake outputs such as
  `nixosModules`, `homeManagerModules`, `profiles`, `overlays`, and `packages`.
- Do not make downstream repos import internal paths from this repo.
- Keep shared profiles user-agnostic: do not hardcode `ysun`, company users,
  company hosts, secrets, LAN addresses, or business services.
- Keep GUI, desktop, input-method, audio, and personal Home Manager behavior in
  personal host modules unless it is intentionally exported as a reusable module.
- Keep company-specific host assembly, `116` disk layout, `/data1`, `jarvis*`,
  Docker service definitions, and company secrets in `sctmes/dotfiles`.
- The `ai-serving` profile must remain host GPU/container support only:
  NVIDIA driver/runtime and Docker GPU plumbing are acceptable; CUDA userspace,
  model runtimes, vLLM, PyTorch, Whisper, SearXNG, and model-service containers
  belong downstream.
- Before editing exported profiles, consider whether the change would surprise a
  downstream consumer. If it would, put the behavior behind a separate exported
  module or leave it in the concrete host.

## Key Constraints

- The repo's shell workflow is Nushell-first. Prefer Nushell syntax when writing or updating repo commands and examples.
- When executing Nushell snippets through Codex tools, do not rely on the tool's shell selection alone. Invoke Nushell explicitly as `nu -c '...'`, otherwise the command may still be interpreted by `/bin/sh`.
- Secrets must go through sops-nix. See `docs/secret-management.md`.
- `vm-test` is only an evaluation / regression target. Do not treat it as a
  real host with product requirements, and do not optimize user-facing changes
  around VM behavior. Keep it evaluating successfully, but gate real hardware,
  desktop, GPU, or personal applications away from it when needed.
- Changes are verified manually by the user after rebuild.
- System rebuild command:
  ```nushell
  sudo nixos-rebuild switch --flake .#homePC
  ```
